# 需求分析技能

## 需求文档模板

### PRD (Product Requirement Document) 结构

```markdown
# [功能名称] 产品需求文档

## 1. 需求背景
- 业务背景和动机
- 要解决的核心问题
- 预期业务价值

## 2. 目标用户
- 用户画像
- 使用场景

## 3. 用户故事

### US-001: [故事标题]
- **作为** [角色]
- **我希望** [功能]
- **以便** [价值]

**验收标准:**
- [ ] AC1: Given [前提] When [操作] Then [结果]
- [ ] AC2: ...

### US-002: ...

## 4. 非功能需求
- 性能: 响应时间 < Xms, QPS > Y
- 安全: 认证、授权、数据加密
- 可用性: SLA 要求

## 5. 优先级
| 用户故事 | 优先级 | 预估工作量 | 备注 |
|---------|--------|-----------|------|
| US-001  | P0     | 3d        |      |

## 6. 开放问题
- [ ] 待确认的问题
```

## 用户故事编写指南

好的用户故事符合 **INVEST** 原则:
- **I**ndependent - 独立，不依赖其他故事
- **N**egotiable - 可协商，不是合同
- **V**aluable - 有价值，对用户有意义
- **E**stimable - 可估算，不会过大
- **S**mall - 小的，一个迭代内可完成
- **T**estable - 可测试，有明确的验收标准

## 验收标准格式 (Gherkin)

```gherkin
Given [初始状态/前提条件]
When [用户执行的操作]
Then [预期结果]
And [可选的额外结果]
```

## 优先级标准

| 级别 | 含义 | 标准 |
|------|------|------|
| P0   | 必须有 | 核心功能，无此功能产品无法使用 |
| P1   | 应该有 | 重要功能，影响用户体验 |
| P2   | 可以有 | 锦上添花，有则更好 |
| P3   | 将来再说 | 远期规划 |
